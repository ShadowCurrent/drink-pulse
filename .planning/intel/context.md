## Topic: Architecture (layers, patterns, boundaries)
- source: docs/architecture.md
- SwiftUI + SwiftData iOS app, all logic on-device; CloudKit sync layered on SwiftData, no custom backend.
- Folder layout: `Domain/`, `Features/{Shell,Dashboard,AddDrink,History,Insights,Settings}`, `Services/`, `DesignSystem/`, `drinkpulseApp.swift`.
- MVVM: views own `@State` + `@Query`; simple mutations via `@Environment(\.modelContext)`; view models `@Observable @MainActor`, receive `[ConsumptionEvent]`/`UserProfile?` as plain injected values, never own a `ModelContext`; domain models are pure SwiftData entities (see ADR-0004 in decisions.md).
- Services layer: platform-capability wrappers behind protocols, thin real adapter + injected fake for tests (ADR-0008). Members: `ReminderService` (+ `NotificationScheduling`); `HealthService` (+ `HealthWriting`, `HKHealthStore` adapter, `UITestHealthStore` stub), shared via `@Environment(\.healthService)`, gated hooks in `HealthWriteHooks` (plan-0036, ADR-0011).
- State management table: view-owned local state → `@State private var`; injected `@Observable` VM needing bindings → `@Bindable var`; read-only injected value → `let`; shared app-wide state → `@Observable` class via `@Environment`. Never `ObservableObject`/`@Published`/`@StateObject`/`@ObservedObject`.
- Navigation: root gate on `@AppStorage("dp_onboarding_done")`; `RootShellView` is a `TabView` with value-based `Tab {}` syntax (Liquid Glass on iOS 26), also guards `UserProfile` existence and resets onboarding if the store is empty; per-tab `NavigationStack`; sheets own dismiss via `@Environment(\.dismiss)`.
- DI: lightweight manual DI via SwiftUI environment values (`@Environment(\.modelContext)`, custom `@Entry` keys). No third-party DI framework.
- Concurrency: Swift 6 strict concurrency; all `@Observable` view models `@MainActor`; structured concurrency for async work; SwiftData ops on the main actor via `ModelContext`, `@ModelActor` for heavy queries when needed.
- Persistence bootstrap: `Domain/Persistence/StoreBootstrap.swift` owns `ModelContainer` creation via an explicit `MigrationPlan` (`schemas = [SchemaV1, SchemaV2, SchemaV3]`, `stages = [v1ToV2, v2ToV3]`); `SchemaV1`/`SchemaV2` are frozen self-contained snapshots, `SchemaV3` references the live `@Model` classes (current shape: `timestamp` renamed `consumptionDate`, added `creationDate`). Snapshot-on-divergence rule (ADR-0009) is called out as having already been violated once in practice ("This bit us once between the first V2 and the rename/creationDate change") — the fix was freezing V2 and adding V3 + `v2ToV3`; states "the next divergence freezes V3 first."
- Non-destructive recovery: on `ModelContainer.init` failure, store files are moved (not deleted) to `Application Support/RecoveredStores/` (max 3 snapshots retained); `clearRecoveredStores()` runs on "Delete all data". With the migration plan in place, this is now strictly the genuine-corruption last resort.
- Record identity / de-dup (plan-0023 Phase A): `ConsumptionEvent`/`DrinkTemplate` carry stable `uuid` (not `.unique`) + `modifiedDate` LWW clock; `UserProfileStore` enforces the single-profile invariant; `RecordDeduplicator` sweeps by uuid on launch/post-sync; CloudKit itself stays OFF pending Phase B (separately approved).
- Data transfer: `Domain/DataTransfer/` — `ExportBundle` (v1 events-only, v2 + optional `ProfileRecord`, future unknown versions throw); `ProfileRecord`; `BackupExport`; `BackupDocument` (`FileDocument` + `.fileExporter`, encode off-main-actor); `DataImporter` (decodes v1/v2, dedupes events, upserts profile, typed `ImportError`).
- Sync: CloudKit via SwiftData's `ModelConfiguration`, no custom sync code, default last-write-wins conflict resolution.

## Topic: Domain rules (calculations, units, entities)
- source: docs/domain.md
- ABV stored as a plain fraction (0.0–1.0); display multiplies by 100.
- `alcoholGrams = volumeMl × quantity × abv × density`; `volumeMl` is a single portion, `quantity` is the count logged in one entry (never folded into volume); never store the derived mass.
- Density depends on display mode AND selected guideline (`AlcoholUnit.density(for:)`, attributed to ADR-0006 amending ADR-0005) — this document reflects the ADR-0006 two-case model (`grams`, `standardDrinks`), which is consistent with ADR-0006 being the sole locked/authoritative density rule (ADR-0005 has since been formally marked Superseded by ADR-0006). See INGEST-CONFLICTS.md INFO (auto-resolved).
- Physical mass always uses 0.789 g/ml (`physicalDensityGramsPerMl`); calories always use it; BAC (when added) must also use it.
- UK NHS units formula: `units = volumeMl × abv% / 1000` (1 unit = 10 ml pure ethanol); 500 ml @ 5% = 2.5 UK units; with the 0.8 display density, 1 UK unit = 8.0 g.
- BAC (Widmark, not yet implemented): `BAC‰ = pureAlcoholGrams / (bodyWeightKg × r × 10) − eliminationRate × hoursElapsed`; r = 0.68 male / 0.55 female; eliminationRate ≈ 0.15‰/hour; display ‰ in EU builds, % BAC in US builds, never mixed; always labeled an estimate. Explicitly: "Do not implement BAC without explicit confirmation."
- Volume units are display-only: `volumeMl` is canonical/exact; `UserProfile.unitSystem` only governs display + which serving presets are offered, never stored data/grams/calories/BAC/guideline%/risk.
- Conversion constants: 1 US fl oz = 29.5735 ml; 1 imperial fl oz = 28.4131 ml.
- Rounding policy: `.metric` → whole ml; `.usCustomary`/`.imperial` → fl oz to one decimal (`formatVolume`, the raw renderer).
- Serving-label rule (plan-0031, hand-verified): `servingVolumeLabel` (serving-list/picker renderer) adopts pint mode for imperial; `isRoundServing(ml)` decides whether an inline ml hint is appended.
- Region-tag policy (plan-0031, reverses plan-0030): R (round-native) / M (real measure) / X (cross-borrow) tagging for serving presets; per-region display names via `VolumeOption.regionNames`.
- Entities: `DrinkTemplate` (reusable preset, `deleteRule: .nullify` so template edits never alter past events, stable `uuid` + `modifiedDate`); `ConsumptionEvent` (fields: `uuid`, `modifiedDate`, `consumptionDate` — renamed from `timestamp`, `creationDate`, `volumeMl`, `quantity`, `enteredUnit`, `price`, `priceCurrency`, `notes`, `location`, `healthKitUUID`); `UserProfile` (singleton, no longer `.unique`, enforced by `UserProfileStore`; fields `bodyWeightKg`, `biologicalSex`, `dateOfBirth`, `guidelineChoice`, `weeklyGoalGrams`, `unitSystem`, `currency`, `abvPrecisionPermille`, `alcoholUnit` default `standardDrinks`, `modifiedDate`). Guideline thresholds are computed on the fly (`GuidelineChoice.limits(for:)`), never stored.
- Drink categories: `beer`, `wine`, `champagne`, `cider`, `alcopop`, `spirits`, `brandy`, `cognac`, `vodka`, `whiskey`, `tequila`, `shot`, `liqueur`, `cocktail`, `fortifiedWine`, `hotDrink`, `custom`.
- Risk level (`RiskLevel.from(pct:)`): `.safe` pct < 0.5, `.caution` 0.5 ≤ pct ≤ 1.0, `.exceeded` pct > 1.0 — the 100% boundary is `.caution`, not `.exceeded`, by design.
- Guideline thresholds table (WHO, DE/DHS, UK/NHS, US/NIAAA, AU/NHMRC 2020, CA/Health Canada LRDG-2011), sex-differentiated where specified; UK weekly = 112 g; AU daily/weekly are independent published caps, not derived from each other; CA std drink = 13.45 g (341 ml × 5% × 0.789).
- Backup/restore format: versioned `ExportBundle` (current version 2: events + optional profile); import behaviour table by version; profile upsert overwrites all fields if a profile exists; export regenerates on a content-signature change (hash over event + profile fields).

## Topic: Product vision & scope
- source: docs/product.md (classified PRD — requirements extracted separately to requirements.md; this entry is the surrounding vision/scope prose)
- Vision: calm, private alcohol tracker, no accounts/subscriptions/cloud sign-in; shows what/when/how compared to health guidelines.
- Privacy stance: no account ever; on-device data, opt-in iCloud sync via the user's own Apple ID; no analytics/crash reporters/third-party SDKs; no AI features that send data off-device.
- Target platforms: Phase 1 iPhone (primary); Phase 2 iPad, Apple Watch complication; minimum deployment iOS 26.
- Note: `bodyWeightKg` exists in `UserProfile` but is not yet surfaced in Settings UI — scaffolded for a future BAC feature.

## Topic: ADR conventions
- source: docs/decisions/README.md
- When to write an ADR: choosing between non-obvious approaches, deviating from a framework convention, accepting a known trade-off, or deciding not to do something. File naming: `NNNN-short-kebab-title.md`. Provides the ADR template (Context / Decision / Consequences / Positive / Negative / Alternatives considered). Status values: Proposed (under discussion), Accepted (in effect), Deprecated (no longer in effect, superseded ADR noted), Superseded (replaced by ADR NNNN). Relevant to the (now-resolved) ADR-0005/ADR-0006 situation in INGEST-CONFLICTS.md: per this table, only a status change to "Superseded" (or "Deprecated") retires an ADR — ADR-0005's Status header has since been updated to "Superseded by ADR-0006" (Superseded 2026-07-18), so it correctly reads as non-locked in this ingest.
