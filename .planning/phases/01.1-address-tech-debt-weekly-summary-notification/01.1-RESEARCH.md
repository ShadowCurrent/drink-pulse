## Summary

This phase has exactly one code change: add a `cancel()` call to `HealthStep`'s weekly-summary toggle-off branch, plus a unit test proving that call fires. There are no new external dependencies, no new packages, and no domain/calculation changes — this is pure internal consistency work inside a single SwiftUI onboarding file plus one new test file.

**The corrected file path:** `HealthStep.swift` actually lives at `Features/Onboarding/Components/HealthStep.swift` — **not** `Features/Onboarding/Steps/HealthStep.swift` as referenced in the phase brief. There is no `Features/Onboarding/Steps/` directory in this codebase; `Features/Onboarding/Components/` is correct. The planner must use this path.

**Primary recommendation:** Make `WeeklySummaryService` constructor-injectable into `HealthStep` (a stored property with a default value, mirroring `WeeklySummaryService`'s own already-existing `init(center:defaults:)` test seam — **not** `@Environment`), and extract the toggle-off action into a small `internal` (not `private`) `async` method so a unit test can `await` it directly without racing a fire-and-forget `Task{}`. This reuses two patterns the codebase already has (constructor injection per ADR-0008, and the `enableWeeklySummary(generation:)` async-method-behind-a-Task shape already in the same file) rather than introducing `@Environment` injection, which does not fit this specific case (see below).

## Open Question Resolution (D-03)

**Question:** Does `HealthStep` need `WeeklySummaryService` injected via `@Environment` (mirroring `healthService`) to make D-03's assertion testable, or is the existing `UITestNotificationCenter` pattern sufficient without changing injection style?

**Answer: Neither, exactly as posed. Use constructor injection (a stored property with a default), not `@Environment` — and pair it with a small visibility change so the test can `await` the call directly.**

### Evidence gathered

1. **`@Environment(\.healthService)` is not the right template to mirror** — `HealthServiceEnvironment.swift` [VERIFIED: local source] states explicitly *why* `HealthService` uses `@Environment`: "Provided once at the app root... so the Settings toggle, the onboarding opt-in, and the Add/Edit/Delete write hooks all read the SAME instance — its per-event serialization... only holds within a single instance." `WeeklySummaryService` has no such shared-instance invariant: `WeeklySummarySection` (`Features/Settings/Components/WeeklySummarySection.swift:26`) already does `private let service = WeeklySummaryService()` — its own independent instance, not environment-shared — and this works correctly today because `cancel()`/`scheduleIfEnabled()` operate against process-wide shared resources (`UNUserNotificationCenter`'s pending-request table, `UserDefaults`) regardless of which `WeeklySummaryService` instance calls them. Adding `@Environment` to `HealthStep` only, while `WeeklySummarySection` keeps inline instantiation, would introduce a *new* asymmetry in the other direction, not resolve one. `[VERIFIED: local source]`

2. **`UITestNotificationCenter` alone does not close the gap** — `UITestNotificationCenter.swift` [VERIFIED: local source] is a launch-argument-gated (`-dp_uitest`) stub selected only inside `WeeklySummaryService.defaultCenter()`. It tracks `pending` state **inside the app process**. XCUITest runs in a **separate process** and has no API to read that in-memory array — the exact same cross-process limitation D-01 already documents for ENGG-07. So a UI test can prove the toggle flips visually (already covered by `OnboardingWeeklySummaryUITests.swift`) but cannot assert "`cancel()` was called" at the object level. `[VERIFIED: local source, cross-referenced with D-01's own reasoning]`

3. **No View-inspection library exists in this project** — confirmed no `ViewInspector` or `SnapshotTesting` dependency anywhere in the repo (checked `.xcodeproj`, no `Package.resolved` with third-party deps). `[VERIFIED: repo search]`

4. **`@testable import` does not expose `private` members**, even within the same module/target — a Swift-wide language rule, not project-specific. `HealthStep.weeklySummaryToggleBinding` and `enableWeeklySummary(generation:)` are currently `private`, so a test file (`drinkpulseTests`) cannot call them even with `@testable import drinkpulse`. `[CITED: SwiftLee — "Testing private methods and variables in Swift"; Swift Forums pitch thread confirms @testable does not override private/fileprivate]`

5. **SwiftUI `View` is now fully `@MainActor`-isolated as a whole protocol** (WWDC 2024 change, active on this project's Xcode 26.6 / iOS 26 minimum-deployment SDK) — so `HealthStep`'s own instance methods are implicitly `@MainActor`, matching `WeeklySummaryService`'s `@MainActor final class` isolation. This is why the existing `enableWeeklySummary()` method already calls `await healthService.requestAuthorization()` and `await WeeklySummaryService().requestAuthorization()` without any extra isolation annotation, and it already compiles clean under this project's zero-warnings strict-concurrency gate (Phase 1 shipped `Complete`, audit status `COMPLIANT`). Mirroring this exact shape for the new `cancel()` call carries no new concurrency risk. `[VERIFIED: local source — existing code compiles today under the same rule; CITED: fatbobman.com "SwiftUI Views and @MainActor" for the WWDC24 @MainActor-on-View-protocol change]`

### Concrete recommendation (what the plan should do)

1. **Give `HealthStep` a constructor-injectable `WeeklySummaryService`** (stored property, default value — same shape as `WeeklySummaryService.init(center:defaults:)`'s own test seam, NOT `@Environment`):
   ```swift
   struct HealthStep: View {
       let onDone: () -> Void
       private let weeklySummaryService: WeeklySummaryService

       init(onDone: @escaping () -> Void, weeklySummaryService: WeeklySummaryService = WeeklySummaryService()) {
           self.onDone = onDone
           self.weeklySummaryService = weeklySummaryService
       }
       // ...
   }
   ```
   Replace the existing inline `WeeklySummaryService()` call in `enableWeeklySummary()` (line ~161) with `weeklySummaryService` too, so both the enable and (new) cancel paths use the same injected instance.

2. **Extract the toggle-off action into a small `internal` (not `private`) `async` method**, mirroring the existing `enableWeeklySummary(generation:)` shape in the same file, so a test can `await` it directly instead of racing a fire-and-forget `Task {}`:
   ```swift
   // Widened from `private` to `internal` (package-default) specifically so
   // drinkpulseTests can call it directly via @testable import and await
   // completion, rather than polling a detached Task.
   func disableWeeklySummary() async {
       weeklySummaryEnabled = false
       weeklySummaryPermissionDenied = false
       await weeklySummaryService.cancel()
   }
   ```
   The binding's `else` branch becomes:
   ```swift
   } else {
       Task { await disableWeeklySummary() }
   }
   ```
   This exactly matches D-02's mandated behavior (mirrors `WeeklySummarySection`'s `Task { await service.cancel() }` off-branch) while making the actual assertion race-free and directly awaitable in a test.

3. **New unit test file** (mirrors CLAUDE.md's "Test organization" rule — production file is `Features/Onboarding/Components/HealthStep.swift`, so the test belongs at):
   `drinkpulseTests/Features/Onboarding/Components/HealthStepTests.swift`
   ```swift
   import Testing
   @testable import drinkpulse

   @MainActor
   struct HealthStepTests {
       @Test func disableWeeklySummary_callsServiceCancel() async {
           let fake = FakeNotificationCenter() // declared in ReminderServiceTests.swift, internal-visible
           let service = WeeklySummaryService(center: fake, defaults: UserDefaults(suiteName: "test.\(UUID().uuidString)")!)
           let step = HealthStep(onDone: {}, weeklySummaryService: service)

           await step.disableWeeklySummary()

           #expect(fake.removedBatches.last == [WeeklySummaryService.weeklySummaryIdentifier])
       }
   }
   ```
   This reuses `FakeNotificationCenter` (declared with no access modifier — internal — in `ReminderServiceTests.swift`, already reused as-is by `WeeklySummaryServiceTests.swift`) exactly as D-03 requires ("reuse that pattern rather than inventing a new one"). `[VERIFIED: local source]`

**Why this is minimal and low-risk:** it touches one production file (`HealthStep.swift`) with a signature-widening + one-method-extraction change, adds one new test file, and does not touch `WeeklySummarySection`, `WeeklySummaryService`, or any Settings code. `WeeklySummaryService`'s own public surface is unchanged.

### Pattern: Constructor injection over `@Environment` for non-shared services
**What:** Give a View a stored property + default-valued init parameter for a service, rather than reaching for `@Environment`.
**When to use:** When the service has no cross-consumer shared-state requirement (unlike `HealthService`, which explicitly needs one instance for serialization). `WeeklySummaryService` has no such requirement — every existing call site (`WeeklySummarySection`, `HealthStep`) already creates its own instance.
**Example:**
```swift
// Source: existing project pattern, WeeklySummaryService.init (Services/WeeklySummaryService.swift:27-33)
init(
    center: NotificationScheduling = WeeklySummaryService.defaultCenter(),
    defaults: UserDefaults = .standard
) {
    self.center = center
    self.defaults = defaults
}
```
Apply the identical shape one level up, at `HealthStep`'s own init, for the service itself.

### Anti-Patterns to Avoid
- **Adding `@Environment(\.weeklySummaryService)` "for consistency with `healthService`":** Looks symmetric but is not justified by any actual shared-instance requirement, and leaves `WeeklySummarySection` on the old inline-instantiation pattern — creating a new inconsistency while "fixing" one that doesn't functionally exist.
- **Testing via `UITestNotificationCenter` state from an XCUITest:** Impossible — XCUITest runs out-of-process; the stub's `pending` array lives inside the app process and has no exposed inspection API. This is the same limitation already documented for D-01/ENGG-07 — don't attempt to bypass it here.
- **Leaving `weeklySummaryToggleBinding`'s `else` branch as a bare closure with a fire-and-forget `Task {}` and trying to unit-test it anyway:** A test cannot reach a `private` computed property's closure even via `@testable import` (Swift's `private` is file-scoped and `@testable` does not lift it), and even if it could, asserting immediately after firing a detached `Task` races the assertion against the awaited `cancel()` call. Extract to a named, awaitable, `internal` method instead.

## Common Pitfalls

### Pitfall 1: Racing a fire-and-forget `Task {}` in a test
**What goes wrong:** If the toggle-off logic stays as `Task { await service.cancel() }` inside a Binding setter and the test asserts immediately after triggering the setter, the assertion runs before the `Task` has a chance to execute — flaky, order-dependent failures.
**Why it happens:** `Task {}` schedules work without the caller awaiting completion; SwiftUI Binding setters are synchronous by protocol requirement, so they cannot themselves be `async`.
**How to avoid:** Extract the `async` logic into a named method the *test* can `await` directly (bypassing the outer `Task {}`, which exists only to satisfy the Binding's synchronous signature) — see Code Examples.
**Warning signs:** A new test that sometimes passes, sometimes fails with `fake.removedBatches` empty.

### Pitfall 2: Assuming `@testable import` exposes everything
**What goes wrong:** Writing a test that tries to call `step.weeklySummaryToggleBinding` or `step.enableWeeklySummary()` directly and getting a compile error ("... is inaccessible due to 'private' protection level").
**Why it happens:** `@testable import` only promotes `internal` → effectively public for the test target; it never lifts `private`/`fileprivate`.
**How to avoid:** Widen only the minimum surface needed (one method) to `internal` (the language default — no keyword needed, just drop `private`), keeping everything else `private`.
**Warning signs:** Build errors referencing "inaccessible due to 'private' protection level" in the new test file.

### Pitfall 3: Conflating this phase's fix with a bigger `HealthStep` refactor
**What goes wrong:** Over-scoping into extracting all of `HealthStep`'s logic into a view model, changing `healthService`'s injection style too, or "fixing" ENGG-07/ENGG-02 alongside this.
**Why it happens:** Once inside the file, related-looking asymmetries (e.g., `healthService` via `@Environment` vs. `weeklySummaryService` via constructor) can look like they need to be unified.
**How to avoid:** CONTEXT.md D-01 and D-04 already close ENGG-07 and ENGG-02 as no-action; stick to the one-line D-02 fix plus the minimal testability seam from this research. Do not touch the Health toggle's own injection style — it has a documented, different reason (`HealthServiceEnvironment.swift`'s single-shared-instance requirement) to use `@Environment`.
**Warning signs:** Diff touching files outside `HealthStep.swift` + its new test file.

### Full before/after for `HealthStep.swift`'s toggle-off path

```swift
// Source: existing project pattern (drinkpulse/Features/Onboarding/Components/HealthStep.swift)
// BEFORE (current, lines ~116-130):
private var weeklySummaryToggleBinding: Binding<Bool> {
    Binding(
        get: { weeklySummaryEnabled },
        set: { newValue in
            weeklySummaryToggleGeneration += 1
            if newValue {
                let generation = weeklySummaryToggleGeneration
                Task { await enableWeeklySummary(generation: generation) }
            } else {
                weeklySummaryEnabled = false
                weeklySummaryPermissionDenied = false
            }
        }
    )
}

// AFTER (D-02 fix + D-03 testability seam):
private var weeklySummaryToggleBinding: Binding<Bool> {
    Binding(
        get: { weeklySummaryEnabled },
        set: { newValue in
            weeklySummaryToggleGeneration += 1
            if newValue {
                let generation = weeklySummaryToggleGeneration
                Task { await enableWeeklySummary(generation: generation) }
            } else {
                Task { await disableWeeklySummary() }
            }
        }
    )
}

/// (`Task { await service.cancel() }`) for consistency (tech-debt item 2,

func disableWeeklySummary() async {
    weeklySummaryEnabled = false
    weeklySummaryPermissionDenied = false
    await weeklySummaryService.cancel()
}
```
