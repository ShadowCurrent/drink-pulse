<objective>
Close tech-debt item 2 from the v1.1 milestone audit (`.planning/v1.1-MILESTONE-AUDIT.md`):
`HealthStep`'s onboarding weekly-summary toggle-off branch does not call
`WeeklySummaryService.cancel()`, unlike `WeeklySummarySection`'s Settings
toggle-off branch. Per CONTEXT.md **D-02**, add the missing `cancel()` call,
and per **D-03**, prove it with an automated service-call assertion test
(not just "existing UI tests keep passing") — reusing the existing
`FakeNotificationCenter` mock-center pattern already used by
`WeeklySummaryServiceTests.swift`, per D-03's explicit instruction not to
invent a new mocking approach.

This is the ONLY code change in Phase 01.1. Tech-debt items 1 and 3 from
the audit are explicit **no-action** decisions this phase does not touch:
- **D-01**: ENGG-07's warm-launch notification-tap path stays untested
  (rated WARNING not BLOCKER by the audit; closing it would need a new
  test-only trigger UI element, declined as out of scope).
- **D-04**: ENGG-02's onboarding "appears inert until next foreground"
  behavior stays as-is (intentional, mirrors `ReminderService`'s existing
  foreground-reschedule convention; confirmed not a bug by the user).

No REQ-IDs are newly minted — ENGG-01 through ENGG-07 stay mapped to
Phase 1 in `.planning/REQUIREMENTS.md` and are unaffected by this fix.

Purpose: eliminate a documented behavioral-consistency gap between the two
places (`Settings`, `Onboarding`) that can toggle the weekly-summary
notification off, before it becomes a real bug (today it is harmless only
because onboarding never calls `scheduleIfEnabled`).

Output: `HealthStep.swift` gains a constructor-injectable
`weeklySummaryService` property and an `internal async
disableWeeklySummary()` method wired to the toggle-off branch;
`HealthStepTests.swift` is a new test file asserting the `cancel()` call
fires.
</objective>
