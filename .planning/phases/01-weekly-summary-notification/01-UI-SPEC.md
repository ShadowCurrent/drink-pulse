## Scope Note

DrinkPulse is a native SwiftUI iOS 26 app (CLAUDE.md: SwiftUI only, no UIKit,
Dynamic Type up to AX5 mandatory). There is no web design-system tooling
(shadcn / Tailwind / component registries) â€” the **shadcn gate does not
apply** to this project. This contract instead documents the project's
existing native `DesignSystem/` tokens and the exact `Features/Settings` +
`Features/Onboarding` component patterns this phase must mirror, per
CONTEXT.md D-05/D-07 ("mirror the existing pattern, don't invent a new
one") and RESEARCH.md's confirmation that every visual pattern this phase
needs already ships in the codebase.

This phase touches exactly **three** real UI surfaces â€” no new screen is
created:

1. **Settings â†’ new `WeeklySummarySection` card** (mirrors `ReminderSection`)
2. **Onboarding â†’ `HealthStep` gets a second, independent toggle row** (mirrors the existing Health toggle block, D-05)
3. **Notification banner copy** (title + 5 body variants â€” the only genuinely new content in this phase; no notification "screen" exists to design, per Apple's fixed banner chrome)

The **tap destination** (Insights tab) requires zero visual changes â€” D-04
locks "open Insights at its default state," so no deep-link UI, no new
component. It is listed here only as an interaction contract, not a visual
one.

**Visual hierarchy:** neither edited screen gets a new focal point. In
Settings, `WeeklySummarySection` sits as a peer card below `ReminderSection`,
same visual weight, no new anchor. In `HealthStep`, the weekly-summary toggle
row sits below the existing Health toggle at equal visual weight inside the
same panel â€” it does not compete with or displace the panel's primary
control.

## UI Considerations

Ran via `ui-consideration-probe.cjs`. Elements classified: Settings toggle row
(E1) and Onboarding toggle row (E2) initially auto-detected as
`list-collection` â€” a false-positive cue trip on the word "row" â€” confirmed
with the user and overridden to `interactive-control` + `static-content` only
(both are singular controls, not repeatable lists, no async load). Notification
banner (E3) classified `static-content`. Re-running against the corrected
kinds narrowed the applicable set to `overflow` + `long-text` across all three
elements; the taxonomy's `error`/`populated`/`empty`/`zero-one-many`/`loading`/
`partial` categories don't map to `interactive-control`/`static-content` kinds,
but the genuinely-relevant error/populated states (permission-denied, banner
body variants) are already covered below from the researcher's direct
domain analysis, not the heuristic.
