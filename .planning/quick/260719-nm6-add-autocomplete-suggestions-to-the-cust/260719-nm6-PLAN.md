<objective>
Add tap-to-autocomplete suggestions to the "Custom Name" text field on the drink Add
(`DrinkDetailInputView`) and Edit (`EditEventView`) screens. As the user types — from the first
character — a suggestion list appears below the field, populated from the distinct `customName`
values previously logged in `ConsumptionEvent` history, matched case-insensitively against the
current input. Tapping a suggestion fills the field with that exact value.

The two screens already have an identical inline `Section` for the custom-name `TextField`
(`editDrink.customName` header, `editDrink.customNamePlaceholder` placeholder). This plan extracts
that into one new shared component (`CustomNameSuggestionSection`, added to
`Features/History/Components/` alongside the existing `PriceCurrencySection` and
`EditNotesSection` — same cross-feature-shared pattern) and adds the suggestion list to it, so both
screens get the feature identically with no duplicated logic.

Purpose: repeat drink names (e.g. "Craft IPA", "Barolo Riserva") are currently retyped by hand
every time; surfacing the user's own history removes that friction with zero new data model or
network surface (on-device only, reads existing `ConsumptionEvent.customName`).
Output: a pure, 100%-unit-tested filtering function; a shared SwiftUI component wired into both
forms; a UI test proving the real tap-to-fill flow.
</objective>

    `CustomNameSuggestionFilter.suggestions(for:in:limit:)`:
    - Empty query string returns an empty array (no suggestions until at least one character is typed).
    - Case-insensitive substring match: given names in mixed case/order, a lowercase query returns
      only the matching names, sorted alphabetically case-insensitively.
    - Names that are case-insensitive duplicates of each other collapse to a single result.
    - Blank / whitespace-only candidate names are ignored entirely.
    - A candidate that case-insensitively equals the query exactly (the value already fully typed)
      is excluded from its own suggestion list, while a longer name sharing that prefix still
      matches.
    - Results are capped at the `limit` parameter (default 8) once more candidates match than the
      cap allows.
    - A candidate that does not contain the query as a substring is excluded.
    Run the test target for this file first and confirm every new test fails (the type does not
    exist yet).
  </behavior>
  <action>
    Create `CustomNameSuggestionFilter` as a `nonisolated enum` in the `Domain/` layer (no SwiftUI,
    no SwiftData import — plain `Foundation`), mirroring the doc-comment density of
    `DrinkMassCalculator` in `DrinkDetailInputView+Logic.swift`. Implement one static function
    `suggestions(for query: String, in names: [String], limit: Int = 8) -> [String]`:
    - Return `[]` immediately when `query` is empty.
    - Trim each candidate name; drop it if the trimmed result is empty.
    - Deduplicate case-insensitively (lowercased trimmed value as the dedup key), keeping the
      first-encountered casing after sorting candidates with `localizedCaseInsensitiveCompare` so
      output order is deterministic and alphabetical.
    - Keep a candidate only when it case-insensitively contains `query`
      (`localizedCaseInsensitiveContains`).
    - Drop a candidate that case-insensitively equals `query` exactly
      (`caseInsensitiveCompare(query) == .orderedSame`) — it is what the user already typed, not a
      suggestion.
    - Stop once `limit` results have been collected.
    Then make the Task 1 tests pass with this implementation; do not add any behavior beyond what
    the tests specify.
