## Summary

This phase's diff is small and disciplined: `HealthStep.swift` gained a constructor-injectable
`weeklySummaryService` property, a two-initializer split (documented, necessary workaround for a

test (`HealthStepTests.disableWeeklySummary_callsServiceCancel`) correctly proves the `cancel()`
call fires via `FakeNotificationCenter`.

No blocking bugs or security issues were found in the diff itself. However, reviewing the full
content of `HealthStep.swift` (as instructed, not just the changed hunks) surfaced a genuine,
pre-existing correctness gap in the *sibling* Health toggle path that sits right next to the
code this phase just hardened with a generation guard — the asymmetry is worth flagging even
though it wasn't in scope for this phase's task list. The new test also has a test-isolation
defect (writes to `UserDefaults.standard` via `@AppStorage`, uncleaned) that risks flakiness as
the suite grows. Both are Warnings, not Blockers.

## Warnings

### WR-01: `enable()` (Health toggle) has no generation guard, unlike the weekly-summary toggle right next to it

**File:** `drinkpulse/Features/Onboarding/Components/HealthStep.swift:118-168`
**Issue:** `weeklySummaryToggleBinding`/`enableWeeklySummary(generation:)` (lines 134-195) were
explicitly hardened with a `weeklySummaryToggleGeneration` counter so that a slow, in-flight
`requestAuthorization()` call cannot clobber a later "turn it back off" action (the WR-02 pattern,
extensively documented in this same file's comments at lines 42-47 and 179-182). The Health
toggle's own `toggleBinding`/`enable()` (lines 118-168) has exactly the same shape — `Task { await
enable() }` on turn-on, synchronous state reset on turn-off — but carries no equivalent guard.

Concretely: user flips Health write ON → `enable()` starts `await healthService.requestAuthorization()`
→ before it resolves, the user flips the toggle OFF (`enabled = false; permissionDenied = false`
runs synchronously) → the suspended `enable()` call resumes, sees `.authorized`, and unconditionally
sets `enabled = true` — silently re-enabling a health-data write path the user just turned off, with
no further UI feedback that this happened. If authorization instead resolves `.denied` after the user
already turned it off, `permissionDenied = true` also fires needlessly, showing a stale error hint for
an action the user reversed. Given CLAUDE.md's stance that consumption/health data and consent state
are sensitive ("Health data is sensitive... treat every field as such"), silently reinstating a
write-enabled flag against the user's last explicit action is a real (if timing-dependent) bug, not
just a cosmetic one.

This same pattern also exists in `HealthSection.swift` (Settings), so it isn't newly introduced by
this phase — but it is directly adjacent to, and inconsistent with, the fix this phase just applied
to the weekly-summary toggle in the very same file, which makes the omission stand out.

**Fix:** Mirror the same generation-counter pattern already used for `weeklySummaryToggleGeneration`:
```swift
@State private var healthToggleGeneration = 0

private var toggleBinding: Binding<Bool> {
    Binding(
        get: { enabled },
        set: { newValue in
            healthToggleGeneration += 1
            if newValue {
                let generation = healthToggleGeneration
                Task { await enable(generation: generation) }
            } else {
                enabled = false
                permissionDenied = false
            }
        }
    )
}

private func enable(generation: Int) async {
    guard let healthService else {
        enabled = false
        return
    }
    _ = await healthService.requestAuthorization()
    guard generation == healthToggleGeneration else { return }
    switch healthService.authorizationStatus() {
    case .authorized:
        permissionDenied = false
        enabled = true
    case .denied, .notDetermined:
        enabled = false
        permissionDenied = true
    }
}
```
If this is intentionally deferred (e.g., treated as its own tech-debt item like D-01/D-04), it should
be logged in the v1.1 audit rather than left as a silent inconsistency between the two toggles in the
same file.

### WR-02: New test writes to `UserDefaults.standard` via `@AppStorage`, with no isolation or cleanup

**File:** `drinkpulseTests/Features/Onboarding/Components/HealthStepTests.swift:13-21`
**Issue:** `HealthStep.weeklySummaryEnabled` and `.permissionDenied`-adjacent state are declared as
`@AppStorage(AppStorageKeys.weeklySummaryEnabled) private var weeklySummaryEnabled = false`
(`HealthStep.swift:40`) with **no explicit `store:`**, so they always read/write `UserDefaults.standard`
— the real, shared app defaults domain used by the actual app under test, not the isolated
`UserDefaults(suiteName: "test.healthStep.\(UUID().uuidString)")` the test constructs for the
*service's* notification bookkeeping.

`disableWeeklySummary()` sets `weeklySummaryEnabled = false` (line 202), which the test's call at
line 18 therefore executes against `UserDefaults.standard` in the test-host process, with no
`removePersistentDomain` / restore-previous-value cleanup afterward. Today this happens to be
harmless because the value written (`false`) matches the property's own default — but it is an
implicit, undocumented dependency on shared global state. As soon as another test in the suite
(now or later) asserts an initial/default value of `AppStorageKeys.weeklySummaryEnabled` from
`UserDefaults.standard`, or a test is added that exercises the `true` path, this becomes a source
of order-dependent flakiness — exactly the class of test-isolation bug the *service*-level tests
(`WeeklySummaryServiceTests`, `ReminderServiceTests`) already take care to avoid by injecting a
private `UserDefaults(suiteName:)` throughout.

**Fix:** Either avoid touching the `@AppStorage`-backed view struct directly in a unit test (test
`WeeklySummaryService.cancel()` in isolation, as `WeeklySummaryServiceTests` already does, and treat
the `HealthStep` wiring as covered by the UI test), or explicitly reset the domain after the assertion:
```swift
@Test func disableWeeklySummary_callsServiceCancel() async {
    defer { UserDefaults.standard.removeObject(forKey: AppStorageKeys.weeklySummaryEnabled) }
    let fake = FakeNotificationCenter()
    let service = WeeklySummaryService(center: fake, defaults: UserDefaults(suiteName: "test.healthStep.\(UUID().uuidString)")!)
    let step = HealthStep(onDone: {}, weeklySummaryService: service)

    await step.disableWeeklySummary()

    #expect(fake.removedBatches.last == [WeeklySummaryService.weeklySummaryIdentifier])
}
```

## Info

### IN-01: `enableWeeklySummary`'s catch block swallows the error without logging

**File:** `drinkpulse/Features/Onboarding/Components/HealthStep.swift:190-194`
**Issue:** The `catch` branch sets `weeklySummaryEnabled = false` / `weeklySummaryPermissionDenied =
true` but never logs the underlying `error`. The sibling `WeeklySummarySection.enable(generation:)`
(Settings) logs on the equivalent path: `logger.error("Weekly summary enable failed:
\(error.localizedDescription)")`. `HealthStep` has no `Logger` instance at all. This isn't a new
regression from this phase (the `catch` body is unchanged), but it's a real diagnostic gap —
authorization failures (permission denied vs. a transient system error) become indistinguishable in
the field.
**Fix:** Add a `Logger` (mirroring `HealthSection`/`WeeklySummarySection`'s
`Logger(subsystem: "com.drinkpulse.app", category: "HealthStep")`) and log in the `catch` block.

### IN-02: Unit coverage for `HealthStep` is limited to the one new toggle-off path

**File:** `drinkpulseTests/Features/Onboarding/Components/HealthStepTests.swift`
**Issue:** The only unit test in the file exercises `disableWeeklySummary()`. `enable()`,
`enableWeeklySummary(generation:)`'s granted/denied/thrown-error branches, and — most notably — the
generation-guard branches this phase's own comments call out as safety-critical (lines 44-47, 179-182,
191) have zero unit-level coverage. `OnboardingHealthStepUITests` only drives the happy "toggle on,
stub auto-grants" path, so the denied path and the generation race are unverified by any automated
test. This is consistent with the phase being explicitly scoped to D-02/D-03 only (D-01/D-04 are
documented no-action items), so it isn't a regression introduced here — but per CLAUDE.md's testing
rules ("Edge cases... boundary conditions" and "Service logic:... authorization-denied and error
paths... must be tested"), this remains an open gap worth tracking as its own tech-debt item if not
already captured.
**Fix:** If/when `enable()`/`enableWeeklySummary` gain a testability seam (as `disableWeeklySummary`
now has), add tests for: denied-on-first-attempt, thrown-error path, and a generation-mismatch
regression test (toggle on then off before the awaited call resolves; assert state doesn't re-arm).

### IN-03: `disableWeeklySummary()` breaks the file's otherwise-consistent `private` action-method convention

**File:** `drinkpulse/Features/Onboarding/Components/HealthStep.swift:201`
**Issue:** Every other action method in the "MARK: - Actions" section (`enable()`,
`enableWeeklySummary(generation:)`) is `private`; `disableWeeklySummary()` alone drops the modifier
to `internal` purely so `@testable import` can call it directly from the test target. This is a
documented, intentional trade-off (per the phase's plan/summary), but it's worth noting as a
recurring pattern to watch: widening production access control solely for test reachability, rather
than testing through the public `Binding` surface (`weeklySummaryToggleBinding`) or extracting the
side-effecting logic into a separately-testable non-View helper type.
**Fix:** No action required now; if this pattern repeats across more methods in this file, consider
extracting the toggle logic into a plain (non-View) helper type that can be unit-tested without
loosening the View's encapsulation.
