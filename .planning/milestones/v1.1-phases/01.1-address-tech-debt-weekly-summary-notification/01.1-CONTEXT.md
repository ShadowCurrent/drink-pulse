# Phase 01.1: Address tech debt: weekly summary notification - Context

**Gathered:** 2026-07-21
**Status:** Ready for planning

<domain>
## Phase Boundary

Close (or explicitly close out) the 3 tech-debt items flagged by the v1.1
milestone audit (`.planning/v1.1-MILESTONE-AUDIT.md`) for Phase 1's weekly
summary notification feature (ENGG-01–07, all already Complete). This phase
does not add new capability — it disposes of 3 specific known gaps:

1. ENGG-07 warm-launch notification-tap path has no UI test coverage.
2. `HealthStep`'s weekly-summary toggle-off doesn't call
   `WeeklySummaryService.cancel()` (inconsistent with Settings' toggle-off).
3. ENGG-02 onboarding opt-in scheduling defers to next app foreground.

After this discussion, only item 2 is an actual code change. Items 1 and 3
are explicitly disposed of as **no action** — see decisions below. No
Requirements are newly minted for this phase (all 7 ENGG requirements stay
mapped to Phase 1 in `.planning/REQUIREMENTS.md`); this phase has no
REQ-IDs of its own.

</domain>

<decisions>
## Implementation Decisions

### Warm-launch notification-tap test coverage (ENGG-07 gap)
- **D-01:** Leave as documented tech debt — **no fix this phase.** The
  audit rated this a WARNING, not a BLOCKER; the actual ENGG-07 requirement
  (cold-launch tap → Insights) is already covered by
  `WeeklySummaryTapUITests`. `UNNotificationResponse` has no public
  initializer XCTest can construct, and XCUITest runs in a separate process
  from the app, so `NotificationCenter.default.post(...)` from a test
  cannot reach `RootShellView`'s `.task` consumer directly — closing this
  gap would require a new test-only trigger UI element, which was
  explicitly declined as out of scope for this phase.

### Onboarding toggle-off `cancel()` consistency gap
- **D-02:** **Fix it.** Add a `WeeklySummaryService().cancel()` call to
  `HealthStep.weeklySummaryToggleBinding`'s `else` (toggle-off) branch —
  see `Features/Onboarding/Steps/HealthStep.swift` (binding at line ~116;
  current off-branch only sets `weeklySummaryEnabled = false;
  weeklySummaryPermissionDenied = false`). Matches
  `WeeklySummarySection.toggleBinding`'s off-branch
  (`Features/Settings/Components/WeeklySummarySection.swift`, calls
  `Task { await service.cancel() }`). No observable behavior change today —
  `HealthStep.enableWeeklySummary()` never calls `scheduleIfEnabled`, only
  `requestAuthorization` (comment confirms: "brand-new onboarding profile
  has zero `ConsumptionEvent`s... eventual foreground reschedule or a later
  Settings toggle handles real scheduling") — this is a future-proofing /
  consistency fix, not a behavior fix.
- **D-03:** Add a **service-call assertion test** for the `cancel()` call
  (not just "existing UI tests keep passing"). `WeeklySummaryService` is
  already injectable via its `center: NotificationScheduling` initializer
  parameter (see `Services/WeeklySummaryService.swift`) and
  `WeeklySummaryServiceTests.swift` / `UITestNotificationCenter` already
  establish the mock-center pattern used elsewhere in this codebase for
  verifying schedule/cancel calls — reuse that pattern rather than
  inventing a new one. **Open implementation question for
  researcher/planner:** `HealthStep` currently instantiates
  `WeeklySummaryService()` inline (default center) rather than receiving it
  via `@Environment` the way `healthService` is injected
  (`@Environment(\.healthService) private var healthService`) — determine
  whether the assertion needs `HealthStep` to accept an injected service
  (mirroring the `healthService` environment pattern) to be verifiable, or
  whether a UI-level assertion via `UITestNotificationCenter` state is
  sufficient. This is Claude's/the planner's call, not locked by the user.

### Onboarding scheduling UX (ENGG-02 "appears inert")
- **D-04:** **No change.** This is the intentional, existing D-01 decision
  from Phase 1's CONTEXT.md (`.planning/phases/01-weekly-summary-notification/01-CONTEXT.md`)
  — reschedule-on-foreground mirrors `ReminderService.scheduleIfEnabled()`,
  and a first-week onboarding user resolves to `.skip` anyway (no
  prior-week data), so there is no visible discrepancy in practice. No copy
  change, no logic change. Confirmed explicitly by the user as not a bug.

### Claude's Discretion
- Whether `HealthStep` needs `WeeklySummaryService` injected via
  `@Environment` (mirroring `healthService`) to make D-03's assertion
  testable, or whether the existing `UITestNotificationCenter` mock-center
  pattern is sufficient without changing `HealthStep`'s injection style —
  see the open question under D-03.

</decisions>

## Existing Code Insights

### Reusable Assets
- `Features/Settings/Components/WeeklySummarySection.swift` —
  `toggleBinding`'s off-branch (lines ~63-78) is the exact pattern to
  mirror in `HealthStep`: `Task { await service.cancel() }` on toggle-off.
- `Services/WeeklySummaryService.swift` — `cancel()` method already exists
  and is used by `WeeklySummarySection`; `init(center:defaults:)` already
  supports test-double injection via `NotificationScheduling`.
- `drinkpulseTests/Services/WeeklySummaryServiceTests.swift` — existing
  mock-center test pattern for `WeeklySummaryService`; reuse this pattern
  for D-03's assertion rather than inventing a new mocking approach.
- `Services/UITestNotificationCenter.swift` — UI-test-facing
  `NotificationScheduling` stub; may be relevant if the assertion needs a
  UI-test-level (not just unit-test-level) verification path.

### Established Patterns
- `HealthStep.swift` (`Features/Onboarding/Steps/HealthStep.swift`) already
  has a parallel, working example of the injection style in question:
  `@Environment(\.healthService) private var healthService` for the Health
  toggle, vs. inline `WeeklySummaryService()` instantiation for the weekly
  summary toggle (`enableWeeklySummary()`, ~line 155). This asymmetry is
  the root of D-03's open question.
- Toggle-generation guard pattern (`weeklySummaryToggleGeneration`,
  mirrored from `WeeklySummarySection`'s `toggleGeneration`) already
  exists in `HealthStep` and must be preserved — the `cancel()` addition
  goes in the `else` branch of `weeklySummaryToggleBinding`, which does
  not currently touch `weeklySummaryToggleGeneration` (only the `if
  newValue` branch does) — do not add generation-guard logic to the
  cancel() call unless research finds a race it needs to guard against.

### Integration Points
- `Features/Onboarding/Steps/HealthStep.swift` — the only file needing a
  source change for D-02 (`weeklySummaryToggleBinding`'s `else` branch,
  ~line 128).
